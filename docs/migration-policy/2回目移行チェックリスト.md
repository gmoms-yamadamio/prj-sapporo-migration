# 2回目データ移行 作業チェックリスト

> 対象手順書: [2回目移行手順書.md](./2回目移行手順書.md)
> 前回チェックリスト: [1回目移行チェックリスト.md](./1回目移行チェックリスト.md)（本書は同ファイルの構成・記入要領を踏襲する）
> 目的: 移行結果に**過不足・不整合がないこと**を、各フロー・各作業実施時にチェック観点に基づいて確認する
> 記入要領:
> - 各チェック項目について、実施日時・実施者・結果を記入し、判定列に `OK` / `NG` を記載する
> - 「実施日時」列は `YYYY-MM-DD HH:MM` 形式で記入する（自動記録の場合は秒単位まで記録される）
> - `NG` の場合は備考に原因・対応方針・再確認結果を記載する
> - 「期待値・確認方法」列の数式（例: `1.3 = 1.1 - 1.2 - 1.15`）は、実施後に必ず突合すること
> - 「入力方法」列は本チェック表の記入を誰が/どう行うかを示す（詳細は次章「結果の自動入力について」）
> - 各行の「JSON key」（`output/reports/checklist_results.json` に記録される実キー）は、1回目チェックリストと**共通のスクリプトが同じキーに記録する項目は1回目と同一の番号・キー**を用いている。2回目にのみ存在する項目は新規キー（`1.16`／`3.16`／`3.17` 等）を使用する

## 0. 結果の自動入力について

チェック結果は、可能な限り ETL スクリプト実行時に自動入力する。仕組みは [1回目移行チェックリスト.md「0. 結果の自動入力について」](./1回目移行チェックリスト.md#0-結果の自動入力について)と同一（`run_all.py` / `record_manual_result.py` / `parse_sb00092_log.py` / `render_checklist_report.py`）。

> **【2回目特有の注意】結果ファイルはラウンド間で共有・上書きされる**
>
> `output/reports/checklist_results.json`（自動集計結果の生データ）は、1回目・2回目で**同一パス・同一ファイル**を使用する。ETLスクリプト（`generate_member.py`／`generate_order.py`等）は1回目・2回目のどちらを実行しても**同じチェックID（`1.1`〜`3.14`等）に上書き**で記録するため、2回目のETLを実行すると1回目の記録は失われる。
>
> - 2回目ETL実行前に、1回目の `output/reports/checklist_report.md`（または本チェックリストの実施記録）を控えとして保管しておくこと（[2回目移行手順書.md §2](./2回目移行手順書.md#2-前提事前準備) 参照）
> - `render_checklist_report.py` は現時点で「1回目移行チェックリスト.md」の項目タイトル・期待値の文言で固定されている。2回目実行後にこのスクリプトを実行すると、**タイトルは「1回目」のままだが値は2回目の値**が表示されるため、本チェックリスト（本ファイル）側の期待値・文言を正として読み替えること（[2回目移行手順書.md §9 #5](./2回目移行手順書.md#9-確認事項要確認前提) 参照）
> - 会員アドレス帳CSV（`3.16`）・会員ポイントCSV（`3.17`）・前回移行済み注文件数（`1.16`）は2回目のために新規追加したチェックIDであり、1回目実行時にも自動記録されるが（実害はない）、1回目チェックリストには項目として存在しない
