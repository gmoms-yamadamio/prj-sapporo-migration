## 1. 対象範囲

7月初旬に抽出される移行データについて、CEC への投入手順を定義する。

| データ | 処理内容 |
| --- | --- |
| 商品データ | すべて新規登録 |
| 会員データ | すべて新規登録（赤星商店 CEC 会員との突合あり。新規 `C` ／ 既存会員は更新 `U`。**2ファイルに分けて出力**） |
| 注文データ | 注文日が 6/30 以前、かつ出荷完了している注文データを新規登録 |

> **【確定】** 会員アドレス帳・会員ポイントは1回目では移行しない（2回目〔サイト閉鎖時点〕にまとめて移行する）方針で確定。現行サイト稼働中はアドレス・ポイントが変動し続けるため、鮮度の高い状態で確定できる2回目にまとめて移行する。

## 2. 前提・事前準備

- 赤星商店の CEC 会員データを CEC 管理画面からエクスポートし `input/cec/member_export.csv` に配置しておく（会員 CSV の突合に必須。[会員CSV.md](./deliverables/会員CSV.md) 参照）
- SB00092 が使用する移行専用固定コード（決済プロファイルコード／発送元コード／配送方法コード等の `migration-xxxxx`）が CEC 本番マスタに登録済みであることを確認する
- 「出荷完了」の判定条件は `PurchaseOrder.shipDate` が **NULL 以外**（出荷日が設定されている）であることに**確定**（本番データで確認済み。[注文履歴CSV.md「出荷完了の判定条件（確定）」](./deliverables/注文履歴CSV.md#出荷完了の判定条件確定) 参照）
- SB00092 実行に必要な環境変数・S3バケット・DB接続情報（`MIGRATION_DATABASE_URL` 等）が本番相当に設定済みであることを確認する

## 3. 全体フロー概要

```text
7月初旬 移行データ受領
   │

① 会員・商品マスタ登録
   ├─ 会員データ・商品データを管理画面インポート
   └─ CECから member.csv（会員ID⇔メールアドレス対応表）をエクスポート

```

## 4. 7月初旬 移行データ受領

- 旧サイトから生CSVを受領し `input/raw/` に格納する

## 5. ETL処理（フォーマット変換）

受領CSVを弊社指定フォーマットへ変換する。会員CSVの突合・振り分けを最優先で行い、他の成果物は会員CSV確定後に生成する（[作業方針設計書.md §7.1](./作業方針設計書.md#71-処理順序依存関係) 準拠）。

- 「注文日が6/30以前」の抽出は、`PurchaseOrder.orderDate` を基準に ETL スクリプト側でフィルタする（確定）

### 生成物

| 成果物 | 用途 | 配置先 | 抽出条件 |
| --- | --- | --- | --- |
| 会員CSV（新規登録用：`member_import_create.csv`） | CEC管理画面インポート用 | `output/processed/` | 赤星商店 CEC 会員に未登録の会員（パターンA `C`。[会員CSV.md](./deliverables/会員CSV.md)） |
| 会員CSV（更新用：`member_import_update.csv`） | CEC管理画面インポート用 | `output/processed/` | 赤星商店 CEC 会員に登録済みの会員（パターンB `U`。同上） |
| 商品マスタ／販売基本情報／販売条件（Excel） | CEC管理画面インポート用 | `output/products/` | `Products.csv` 全件 |
| 注文CSV／注文明細CSV（`order.csv`／`order_detail.csv`） | SB00092 入力用 | `output/processed/` | 注文日（`PurchaseOrder.orderDate`）が6/30以前、かつ出荷完了（`shipDate` が NULL 以外）、ゲスト・キャンセル・削除済み会員の注文は除外（[注文履歴CSV.md](./deliverables/注文履歴CSV.md)） |

- 件数突合・エラーリストを `output/reports/` に出力し確認する（突合不能会員、除外注文等）

## 6. ① 会員・商品マスタ登録

1. CEC管理画面CSVアップロード機能で以下を登録する
   - 会員データ（新規登録用CSV：`member_import_create.csv`）
   - 会員データ（更新用CSV：`member_import_update.csv`）
   - 商品データ（全件）※会員系と並行実施可
2. **CECから `member.csv`（会員ID⇔メールアドレス対応表）をエクスポート**し、S3の `migration/memberPoint/input/member.csv` に配置する
   - SB00092はこの `member.csv` を入力として使用する（メールアドレス→CEC会員IDの変換に必要）。手順1の会員データ登録が完了していないとエクスポートできないため、その直後に実施する
   - 注意：ここで言う `member.csv` は、手順1でインポートした「会員CSV（管理画面インポート用）」とは**別のファイル**である（CECからのエクスポート結果）
   - この `member.csv` は、2回目以降の会員データ突合（1回目移行済み会員の特定）にも利用できるよう保管しておく（[2回目移行手順書.md](./2回目移行手順書.md) 参照）

## 7. ② バッチ用ファイル配置・バッチ実行

1. バッチ用に以下ファイルをS3にアップロードする
   - `member.csv`（[6.2](#6-会員商品マスタ登録) で取得した会員ID⇔メールアドレス対応表）
   - 注文CSV／注文明細CSV（`order.csv`／`order_detail.csv`、SB00092入力）
2. **SB00092**：注文情報移行バッチを実行（`member.csv` ＋ `order.csv`／`order_detail.csv` を使用）
   - 1回目は会員ポイント移行（SB00091）を実施しないため、`new_member_point.csv` は配置しない
   - `new_member_point.csv` が存在しない場合、SB00092は「会員ポイント紐付け用ダミー注文」処理について一時DBの既存データ（初回実行時は空）をそのまま利用してスキップする（[SB00092注文情報移行バッチ処理概要.md §4.1](./SB00092注文情報移行バッチ処理概要.md#41-入力ファイルと配置)）。ダミー注文は作成されない想定
   - 初回の本番データ投入となるため、環境変数 `MIGRATION_PROCESS_MODE=db` で先にCSV取込・件数確認を行い、問題なければ `api` モードでCEC API実行する段階実行を推奨する

## 8. 完了確認

- SB00092 の処理結果ログ（成功件数／エラー件数）を確認する
- 会員・商品の登録件数を [5.](#5-etl処理フォーマット変換) の突合レポートと照合する
- 移行した注文の件数が、抽出条件（注文日6/30以前・出荷完了）に合致した対象件数と一致するか確認する
