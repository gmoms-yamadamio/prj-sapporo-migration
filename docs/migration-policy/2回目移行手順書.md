## 1. 対象範囲

7/31 に現行サイト（シュパーク）を閉鎖した後に抽出される移行データについて、CEC への投入手順を定義する。

| データ | 処理内容 |
| --- | --- |
| 会員データ | 未登録 → 新規登録／更新済（かつ第1回で移行した会員） → 更新（**新規登録用1ファイル＋更新用2ファイル（B-1／B-2）に分けて出力**） |
| 会員アドレス | すべて新規登録 |
| 会員ポイント | すべて登録（加算） |
| 注文データ | 前回移行していない、かつ出荷完了している注文データを新規登録 |

## 2. 前提・事前準備

- **1回目移行済み会員データで `input/cec/member_export.csv` を更新しておくこと（ETLスクリプトが直接読み込むため必須）**
  - [1回目移行手順書 §6.2](./1回目移行手順書.md#6-会員商品マスタ登録) で取得・保管した `member.csv`（会員ID⇔メールアドレス対応表）を、2回目ETL実行前に `input/cec/member_export.csv` として配置（上書き）する
  - `generate_member.py`／`generate_address.py`／`generate_point.py` はいずれも固定パス `input/cec/member_export.csv` を読み込む（ラウンドによる自動切替なし）。1回目時点の赤星商店会員のみのファイルのまま実行すると、**1回目で移行した会員が再度パターンC（新規重複登録）として出力される**ため、必ず更新すること
  - 上書き前に旧ファイルを `input/cec/member_export_round1.csv` 等に退避しておくと事故防止・切替検証に有用
  - ※2回目の会員CSV生成ロジックは [会員CSV.md](./deliverables/会員CSV.md) の「2回目以降の突合ロジック」に従う（**メールアドレスで CEC 既存会員と突合**）
- **1回目ETL実行時の `output/processed/order.csv` を退避しておくこと（前回移行済み注文の除外判定に必須）**
  - 例: `output/processed/order_round1.csv` にコピーして保管する
  - `output/` はGit管理外であり、かつ2回目ETL実行時に同名の `order.csv` へ上書きされるため、退避を忘れると [5.](#5-731-etl処理フォーマット変換) で前回移行済み注文の除外ができなくなる（詳細は [scripts/README.md「移行ラウンドの切り替え」](../../scripts/README.md#移行ラウンドの切り替え注文抽出ルール) 参照）
  - 同様に `output/reports/checklist_results.json` も2回目ETL実行時に上書きされるため、1回目の実施記録（[1回目移行チェックリスト.md](./1回目移行チェックリスト.md) の実行結果）は事前に `output/reports/checklist_report.md` 等の控えを保管しておくこと
- SB00092 が使用する移行専用固定コード（決済方法グループコード `migration-pay-group` ／発送元コード `migration-warehouse` ／配送方法グループコード `migration-delivery-group`）が CEC 本番マスタに登録済みであることを確認する
- SB00091／SB00092 実行に必要な環境変数・S3バケット・DB接続情報（`MIGRATION_DATABASE_URL` 等）が本番相当に設定済みであることを確認する

## 4. 7/31 移行データ受領

- 現行サイト閉鎖後、旧サイトから生CSVを受領し `input/raw/` に格納する

## 5. 7/31～ ETL処理（フォーマット変換）

受領CSVを弊社指定フォーマットへ変換する。会員CSVの突合・振り分けを最優先で行い、他の成果物は会員CSV確定後に生成する（[作業方針設計書.md §7.1](./作業方針設計書.md#71-処理順序依存関係) 準拠）。

### 実行コマンド（環境変数必須）

ETLスクリプトは既定でラウンド1のルール（注文日6/30以前カットオフ）が適用される。**2回目実行時は以下2つの環境変数を必ず指定すること。**

```bash
MIGRATION_ROUND=2 \
MIGRATION_PREVIOUS_ORDER_IDS_CSV=output/processed/order_round1.csv \
python scripts/run_all.py
```

| 環境変数 | 内容 | 未指定時の影響 |
| --- | --- | --- |
| `MIGRATION_ROUND=2` | 注文抽出ルールを2回目用（カットオフ日付なし・前回未移行・出荷完了のみ対象）に切り替える（`scripts/config/round_rules.json` 参照） | 既定値`1`が適用され、6/30以前の注文しか抽出されない（7/1以降の注文が漏れる） |
| `MIGRATION_PREVIOUS_ORDER_IDS_CSV` | [2.](#2-前提事前準備) で退避した1回目の `order.csv` を指定し、前回移行済み注文を除外する（カンマ区切りで複数指定可） | 除外が一切行われず、1回目移行済み注文がすべて2回目の対象に含まれる（SB00092側の重複チェックは直近1年限定の補助的なものであり、これに依存しない） |

実行前に [2. 前提・事前準備](#2-前提事前準備) の `input/cec/member_export.csv` 更新・`order.csv` 退避が完了していることを確認する。詳細は [scripts/README.md「移行ラウンドの切り替え（注文抽出ルール）」](../../scripts/README.md#移行ラウンドの切り替え注文抽出ルール) 参照。

### 生成物

| 成果物 | 用途 | 配置先 |
| --- | --- | --- |
| 会員CSV（新規登録用：`member_import_create.csv`） | CEC管理画面インポート用 | `output/processed/` |
| 会員CSV（更新用・全項目上書き：`member_import_update_full.csv`） | CEC管理画面インポート用 | `output/processed/` | 当初「新規登録`C`」だった会員で、前回抽出日時以降に更新あり（B-1フォーマット。[会員CSV.md](./deliverables/会員CSV.md)） |
| 会員CSV（更新用・メルマガ配信のみ：`member_import_update_mailmag.csv`） | CEC管理画面インポート用 | `output/processed/` | 当初「更新のみ`U`」だった会員で、前回抽出日時以降に更新あり（B-2フォーマット。同上）。該当会員がない場合はファイル未出力または0件 |
| 会員アドレス帳CSV | CEC管理画面インポート用 | `output/processed/` |
| 会員ポイントCSV（`member_point.csv`） | SB00091 入力用 | `output/processed/` |
| 注文CSV／注文明細CSV（`order.csv`／`order_detail.csv`） | SB00092 入力用 | `output/processed/` |
| 商品マスタ／販売基本情報／販売条件（Excel） | CEC管理画面インポート用（1回目未登録商品のみ抽出） | `output/products/` |

- 件数突合・エラーリストを `output/reports/` に出力し確認する（突合不能会員、除外注文等）
- **商品データの「1回目未登録商品のみ抽出」は完全に手動作業である**。`generate_product_reference.py` は `Products.csv` の全件からSKU参照表（税率判定用）を生成するのみで、1回目登録済み商品との自動差分判定は行わない。担当者が1回目に登録した商品リストと `Products.csv` を目視突合し、新規追加分のみを商品マスタExcelに反映すること

## 6. 8/1（仮）① 会員・商品マスタ登録

1. CEC管理画面CSVアップロード機能で以下を登録する
   - 会員データ（新規登録用CSV：`member_import_create.csv`）
   - 会員データ（更新用CSV・全項目上書き：`member_import_update_full.csv`）
   - 会員データ（更新用CSV・メルマガ配信のみ：`member_import_update_mailmag.csv`）
   - 商品データ（※1回目に未登録の商品のみ）※会員系と並行実施可
2. **CECから `member.csv`（会員ID⇔メールアドレス対応表）をエクスポート**し、S3の `migration/memberPoint/input/member.csv` に配置する
   - SB00091・SB00092は共にこの `member.csv` を入力として使用する（メールアドレス→CEC会員IDの変換に必要）。手順1の会員データ登録が完了していないとエクスポートできないため、その直後に実施する
   - 注意：ここで言う `member.csv` は、手順1でインポートした「会員CSV（管理画面インポート用）」とは**別のファイル**である（CECからのエクスポート結果）
3. CEC管理画面CSVアップロード機能で会員アドレスデータを登録する
   - 手順1の会員データ登録完了後に実施する（メールアドレスで会員と紐付けるため）

## 7. 8/1（仮）② バッチ用ファイル配置・バッチ実行

1. バッチ用に以下ファイルをS3にアップロードする
   - `member.csv`（[6.2](#6-8-1仮-会員商品マスタ登録) で取得した会員ID⇔メールアドレス対応表）
   - 会員ポイントCSV（`member_point.csv`、SB00091入力）
   - 注文CSV／注文明細CSV（`order.csv`／`order_detail.csv`、SB00092入力）
2. バッチ実行
   - **SB00091**：会員ポイント移行バッチを実行 → `new_member_point.csv` 出力
   - **SB00092**：注文情報移行バッチを実行（`member.csv` ＋ `new_member_point.csv` ＋ `order.csv`／`order_detail.csv` を使用）
   - 初回の本番差分データ投入となるため、環境変数 `MIGRATION_PROCESS_MODE=db` で先にCSV取込・件数確認を行い、問題なければ `api` モードでCEC API実行する段階実行を推奨する（一括 `all` 実行よりリスクを抑えられる）
3. CEC管理画面CSVアップロード機能で会員ポイントCSV（SB00091出力：`new_member_point.csv`）を登録する

## 8. 完了確認

- SB00091／SB00092 の処理結果ログ（成功件数／エラー件数）を確認する
- 会員・アドレス帳・商品の登録件数を [5.](#5-731-etl処理フォーマット変換) の突合レポートと照合する
- SB00092 の重複チェック（`custom_old_order_number` 検索）によりスキップされた注文がないか確認する（1回目分との意図しない重複がないか）
