# データ移行ETL 作業方針設計書

## 1. 目的

本ドキュメントは、サッポロビール社 EC サイトリプレイスに伴うトランザクションデータ移行（会員・商品・注文・ポイント）について、**旧サイト抽出データ（CSV）から弊社指定フォーマットの CSV を生成する**作業の全体方針を定義する。

対象リポジトリ／フォルダ: `prj-sapporo-migration/data-migration-etl/`

各成果物の詳細な作成方針は、`deliverables/` 配下の個別設計書を参照すること。

---

## 2. 背景と方針変更

### 2.1 当初方針

- お客様に、会員データの重複排除済みかつ弊社指定フォーマットの CSV を提供いただく
- 弊社は受領 CSV をもとに CEC への移行処理を実施する

### 2.2 今回の方針（変更後）

- お客様側で加工しきれないため、**旧サイトから抽出した生 CSV** を弊社が受領する
- **重複排除・フォーマット変換（ETL）を弊社で実施**し、弊社指定フォーマットの CSV を生成する
- 生成した CSV を、CEC 管理画面インポートまたは既存移行バッチ（SB00091 / SB00092）へ投入する

### 2.3 スコープ

| 区分 | 内容 |
| --- | --- |
| 対象 | 商品マスタ、販売基本情報、販売条件、会員、会員アドレス帳、会員ポイント、注文履歴、注文履歴明細 |
| 対象外 | 法人 CSV（設計書シート「法人 CSV 項目」は今回の成果物に含めない） |
| 対象外（別フォルダ） | コンテンツ移行（`content-migration/` で別途対応） |

| `PurchaseOrder.csv` | 注文ヘッダ | 100 |

| `OrderLine.csv` | 注文明細 | 100 |

### 5.1 テーブル間の関係（暫定）

```text
User (Id) ── UserAccount (UserNo) ── PointBankAccount (UserNo)
                │
                ├── UserAddress → Address
                └── OrderCustomer → PurchaseOrder
                        └── OrderLine
```

### 5.2 入力データの留意点

- 会員 ID 系統が複数存在する（`User.Id` / `UserAccount.UserNo` / `OrderCustomer.UserNo`）
- `User.DeletedAt` に値があるレコードは削除済みアカウント → **移行対象外**
- ゲスト注文あり（`OrderCustomer.isGuest = 1`）→ 注文移行対象外
- 旧 `PurchaseOrder.siteId` に複数値があるが、移行元はシュパーク 1 サイト（出力は一律 `shupark`）
- `Products.csv` は 3 行ヘッダ構成（日本語項目名 / 型定義 / 英語項目名）
- サンプルデータは個人情報がマスク（`***`）されている

### 5.3 赤星商店会員との共用

会員・ポイント・注文の移行では、**クラウド EC 運用中の赤星商店会員データ**（`input/cec/`）との突合が必要。詳細は [会員CSV.md](./deliverables/会員CSV.md) を参照。

## 12. ETL 全体アーキテクチャ（確定版）

2026-07-01 時点で、前段 ETL の実装方針を確定した。

### 12.1 方針サマリー

| 観点 | 決定内容 |
| --- | --- |
| 加工の実装形態 | **スタンドアロン Python スクリプト**（`scripts/`）。移行バッチへの組み込みは行わない |
| 中間 DB の要否 | **前段 ETL では不要**。注文系の DB 投入は SB00092（Prisma）に任せる |
| 赤星商店会員データの取得 | **初回は管理画面 CSV エクスポート** → `input/cec/` 配置。本番前に API 化を検討 |
| 会員 CSV の投入 | **管理画面 CSV インポートのみ**（API 経由の登録・更新は行わない） |
| 会員ポイント・注文 | 前段 ETL は **SB00091/92 入力 CSV を直接生成**。バッチは既存のまま利用 |

### 12.2 全体処理フロー

```text
[input/raw/ 旧サイト生 CSV]
        │
        ├─ Products.csv ──────────────────→ [商品系 Excel] ──→ 管理画面インポート
        │                                  output/products/
        │
        └─ User/Order 系 CSV ──→ [scripts/ Python ETL]
                                      │
                    ┌─────────────────┼─────────────────┐
                    ▼                 ▼                 ▼
              member.csv*      member_point.csv    order.csv
              address.csv                         order_detail.csv
                    │                 │                 │
                    ▼                 ▼                 ▼
              管理画面インポート   SB00091          SB00092
              (会員・アドレス帳)   (メール→会員ID)   (DB→API)
                    │                 │                 │
                    └─→ member.csv ────┴─────────────────┘
                        (会員ID↔メール対応表)
                        input/cec/ または CEC エクスポート

* member.csv（SB00091/92 用）は会員インポート後に CEC からエクスポートして配置
```

### 12.3 成果物ごとの処理経路

| # | 成果物 | ETL 手段 | 出力先 | 投入先 |
| ---: | --- | --- | --- | --- |
| 1〜3 | 商品マスタ / 販売基本情報 / 販売条件 | Excel 関数 | `output/products/` | CEC 管理画面 |
| 4 | 会員 CSV | `generate_member.py` | `output/processed/member_import_create.csv`（新規） / `member_import_update.csv`（更新） | CEC 管理画面 |
| 5 | 会員アドレス帳 CSV | `generate_address.py` | `output/processed/address_import.csv` | CEC 管理画面 |
| 6 | 会員ポイント CSV | `generate_point.py` | `output/processed/member_point.csv` | SB00091 入力 |
| 7 | 注文履歴 CSV | `generate_order.py` | `output/processed/order.csv` | SB00092 入力 |
| 8 | 注文履歴明細 CSV | `generate_order_detail.py` | `output/processed/order_detail.csv` | SB00092 入力 |
| — | member.csv（対応表） | CEC エクスポート（ETL 外） | `input/cec/member.csv` | SB00091/92 入力 |

### 12.4 実行順序（本番移行時）

1. 会員 CSV 生成・管理画面インポート
2. CEC から `member.csv`（会員 ID ↔ メールアドレス）をエクスポート
3. 会員アドレス帳 CSV 生成・管理画面インポート
4. 会員ポイント CSV 生成 → SB00091 実行 → `new_member_point.csv` 生成
