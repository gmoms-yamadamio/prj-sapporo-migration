## 2. 全体アーキテクチャ（共通バッチ基盤）

このリポジトリの全バッチは共通のテンプレート構造に従う。

```text
index.ts (execBatch呼び出し)
  └─ BatchMain.execBatch()
       ├─ initializeConfig()          … .env / Lambda環境変数の読込み、必須環境変数チェック
       └─ BatchFactory.createProcessor()
            └─ BatchTemplate.process()
                 ├─ projectLogic.startProcess()   … MessageBuffer初期化、開始ログ
                 ├─ projectLogic.execute()        … ★ 各バッチの本処理（BaseLogicで実装）
                 └─ projectLogic.endProcess()     … 終了ログ（所要時間出力）
```

- `BatchFactory` はプロジェクトコード（`PROJECT_CODE`）に応じてプロジェクト固有ロジック（`projectLogics/impl/BaseLogic.ts`）があれば動的 import し、無ければ `projectLogics/BaseLogic.ts`（共通ロジック）にフォールバックする。
  - SB00092 では現時点でプロジェクト固有クラスは存在せず、`projectLogics/BaseLogic.ts` の `BaseLogic` がそのまま実行される。
- エラーは `BatchTemplate.process()` の `try/catch` で捕捉され `handleError()` に渡る。エラー発生時も `finally` で `endProcess()`（後処理）が必ず実行される。
- ログは `MessageBuffer`（シングルトン）にメッセージ／エラーを蓄積しつつ、都度 `console.log` / `console.error` に出力する（外部ログ集約先は Lambda/ECS のログドライバ〈CloudWatch Logs 等〉に委譲）。

## 3. 処理の全体フロー

`BaseLogic.execute()` が以下の順に処理する。

```text
execute()
 ├─ preProcess()                      … 前処理
 │    └─ 実行モード判定（MIGRATION_PROCESS_MODE）
 │
 ├─ mainProcess()                     … 本処理
 │    ├─ processDbRegistration()      … [モード db/all] CSV → 一時DB 全削除・全登録
 │    └─ processApiExecution()        … [モード api/all] 一時DB → CEC注文系API実行
 │
 └─ postProcess(success)              … 後処理（結果ログ、Prisma切断）
      ※ 例外発生時は catch → postProcess(false) → 例外re-throw
```

### 処理モード（`MIGRATION_PROCESS_MODE` 環境変数）

| モード | 動作 |
|---|---|
| `db`（既定値は `all`） | CSV 読込 → 一時 DB への全削除・全登録のみ実施 |
| `api` | 一時 DB のデータを使って CEC API 実行のみ実施 |
| `all` | `db` → `api` を続けて実施（本番運用時の標準モード） |

DB登録とAPI実行を分離できるため、「CSV取込は済ませたが投入は後で確認してから」といった段階実行が可能な設計になっている。

## 4. DB 登録処理（`processDbRegistration`）

### 4.1 入力ファイルと配置

S3（本番/staging）またはローカル（開発環境, `ENV_MODE=develop`）から以下 4 種類の CSV を読み込む。ファイルが存在しない場合はその種別の登録処理をスキップし、既存の一時DBデータ（前回実行分）をそのまま利用する。

| 用途 | S3キー（既定パス） | ローカルパス |
|---|---|---|
| 会員CSV（会員ID⇔メールアドレス対応表） | `migration/memberPoint/input/member.csv` | `./tmp/migration/memberPoint/input/member.csv` |
| 新会員ポイントCSV（SB00091出力） | `migration/memberPoint/output/new_member_point.csv` | `./tmp/migration/memberPoint/output/new_member_point.csv` |
| 注文CSV | `migration/orderInfo/input/order.csv` | `./tmp/migration/orderInfo/input/order.csv` |
| 注文明細CSV | `migration/orderInfo/input/order_detail.csv` | `./tmp/migration/orderInfo/input/order_detail.csv` |

- S3ベースパスは環境変数 `MIGRATION_MEMBER_POINT_S3_BASE_PATH`（既定 `migration/memberPoint`）、`MIGRATION_ORDER_INFO_S3_BASE_PATH`（既定 `migration/orderInfo`）で変更可能。
- CSVはヘッダ有無を自動判定（1行目に「注文番号」「会員ID」を含むか）してパースする。

### 4.2 一時DB（Prisma / MySQL, `MIGRATION_DATABASE_URL`）

`prisma/schema_migration.prisma` で定義される3テーブル（`migration-client` として生成）。

| テーブル | 用途 | 主なカラム | `api_registered` |
|---|---|---|---|
| `migration_member_point`（`MigrationMemberPoint`） | 会員ポイント移行のうち、注文バッチが必要とする「会員ID＋メールアドレス」情報 | `member_id`, `email` | API実行済フラグ（0:未登録／1:登録済） |
| `migration_order`（`MigrationOrder`） | 注文ヘッダ移行データ（購入者・受取人情報を含む） | `order_number`, `site_code`, `member_email`, 金額各種, 購入者/受取人 住所・氏名 等 | 同上 |
| `migration_order_detail`（`MigrationOrderDetail`） | 注文明細移行データ | `order_number`, `detail_number`, `sku_code`, `unit_price`, `quantity` 等 | （フラグなし） |

### 4.3 登録手順（テーブル単位で独立）

各CSVが存在する場合のみ、そのテーブルに対して実施：

1. **バリデーション** — 行単位で必須／文字数／半角数字／メール形式／サイトコード列挙値等をチェック（`validateOrderRows` / `validateOrderDetailRows`）。エラー行はログ出力の上スキップし、正常行のみ登録対象とする（バッチ全体は止めない）。
2. **全削除** — `prisma.migrationXxx.deleteMany()` で対象テーブルを全件削除。
3. **全登録** — `createMany` を1,000件単位のチャンクに分割して `insertInChunks` で登録。

会員ポイントCSVについては、会員CSVでメールアドレス→会員IDのMapを作り、`new_member_point.csv` に含まれるユニークな会員IDごとに1レコードを `migration_member_point` に登録する（ポイント数自体はこのテーブルには保持せず、あくまで「対象会員」を洗い出す用途）。

## 5. API 実行処理（`processApiExecution`）

一時DB（`api_registered = 0` のレコード）を対象に、CEC の注文系API（`src/common/services/cecApi/CecApiClient.ts`）を呼び出して本番相当の注文を作成する。大きく **(A) 注文移行データ** と **(B) ダミーデータ** の2種類を処理する。

呼び出す CEC API（`CEC_API_BASE_URL` 配下）:

| API | 用途 |
|---|---|
| `ordersList`（注文一覧取得） | 重複登録防止のための事前チェック |
| `ordersCreateAuction`（オークション注文作成API） | 注文本体の新規作成 |
| `ordersSetStatus`（注文ステータス更新API） | 作成した注文にカスタム項目（旧注文番号等）を設定 |

### (A) 注文移行データの処理フロー

```text
migration_order（api_registered=0）を全件取得
  └─ 1件ごとに:
       1. member.csv から会員メールアドレス→会員ID を引き当て（未取得ならスキップ・エラー計上）
       2. ordersList をカスタム条件 custom_old_order_number=(CSVの注文番号) で検索し重複チェック
            - 既に存在する場合 → api_registered=1 に更新してスキップ（成功扱い）
       3. buildMigrationOrderQuery() で注文本体・購入者・配送・SKU情報を組み立て、
          ordersCreateAuction を実行
       4. 作成成功時、ordersSetStatus でカスタム項目 custom_old_order_number に
          CSVの旧注文番号を設定（新旧注文番号のトレーサビリティ確保）
       5. すべて成功したら migration_order.api_registered = 1 に更新
```

- 注文明細（`migration_order_detail`）は `order_number` をキーに事前にMap化しておき、SKU配列として注文に含める。
- 金額・数量等はCSV上は文字列のため `parseInt` で数値変換（変換失敗時は `0` または `1` にフォールバック）。
- 都道府県名 → CEC都道府県コードへの変換は `constants/mapping/prefecture.ts` を利用。
- 配送方法・発送元・決済方法グループ等は移行専用の固定コード（`migration-warehouse` / `migration-delivery-group` / `migration-pay-group` 等）を使用する（本番マスタ登録値については [注文履歴CSV.md](./deliverables/注文履歴CSV.md) の未確定事項を参照）。

### (B) API 実行処理 - ダミーデータ（会員ポイント紐付け用）

```text
migration_member_point（api_registered=0）を全件取得
  └─ 1件ごとに:
       1. ordersList をカスタム条件 custom_migration_dummy=(会員ID) で検索し重複チェック
            - 既に存在する場合 → api_registered=1 に更新してスキップ（成功扱い）
       2. buildDummyOrderQuery() で金額0円・ダミー購入者情報の注文を組み立て、
          ordersCreateAuction を実行
       3. 作成成功時、ordersSetStatus でカスタム項目 custom_migration_dummy に
          会員IDを設定
       4. migration_member_point.api_registered = 1 に更新
```

> **前提（要確認）**: ソースコード上、SB00091 が生成した会員ポイント対象会員ごとに金額0円のダミー注文を1件作成している。これは「CEC側で会員ポイントの付与・履行に注文実績（`order`）が前提として必要」という業務要件に対応するための仕組みと推測されるが、コード内にその意図を明記したコメントは無いため、**業務要件としての正式な意図は要お客様・開発チーム確認**。

### 処理結果ログ

(A)(B) それぞれについて「成功件数 / エラー件数」を最後に集計してログ出力する（`注文移行データAPI: 成功 n件 / エラー n件` 等）。個別の失敗はスキップされ、バッチ全体は継続する（1件のエラーで全体停止しない設計）。

- `endProcess()`（共通基盤側）で開始からの所要時間をログ出力。

## 8. 前段ETL（本プロジェクト側）との関係

本バッチ（SB00092）は入力として **注文履歴CSV** と **注文履歴明細CSV** を必要とする。これらは `data-migration-etl` 側の Python スクリプト（[作業方針設計書.md §12](./作業方針設計書.md#12-etl-全体アーキテクチャ確定版) 参照）で `output/processed/order.csv` / `order_detail.csv` として生成され、S3の `migration/orderInfo/input/` 配下（または開発時はローカル `tmp/migration/orderInfo/input/`）に配置してSB00092を実行する運用を想定している。

また、会員IDとメールアドレスの対応表（`member.csv`）は、会員CSVをCEC管理画面へインポートした後にCECからエクスポートして `input/cec/` 等に配置する運用（[作業方針設計書.md §12.4](./作業方針設計書.md#124-実行順序本番移行時)）であり、SB00092単体では生成できない外部依存データである点に注意。

実行順序としては、`会員CSVインポート → member.csvエクスポート → SB00091（会員ポイント移行, new_member_point.csv生成） → SB00092（注文移行）` の順で実行する必要がある。
